01 — O que é um Embedding?

Um embedding transforma palavras, frases ou documentos inteiros em uma lista de números (vetor). Esses números capturam o significado semântico — conceitos relacionados ficam matematicamente próximos, mesmo que as palavras sejam diferentes. É assim que o sistema Python de Pelotas lê o protocolo de curativos e entende perguntas clínicas sem precisar de palavras exatas.

"Ferida com esfacelo amarelado, exsudato moderado, bordas descoladas — indicar hidrofibra"

02 — Conceitos do Protocolo Indexados pelo Sistema

O sistema Python indexa o protocolo municipal de Pelotas, enriquecido com o Manual SMS-SP (2021) como referência técnica nacional. Cada conceito abaixo está representado como vetor na base de dados local.

🛏️
Úlceras por Pressão
Estágios I–IV, sem estágio, Escala Braden, mudança de decúbito, espumas preventivas
🩺
Avaliação RYB
Sistema Vermelho/Amarelo/Preto: proteger granulação, limpar esfacelo, desbridar escara
🦠
Infecção e Biofilme
Critérios diagnósticos, carga bacteriana >10⁵ CFU, técnica de Levine, PHMB, prata
🩹
Curativos Padronizados
Hidrofibra Ag, espumas c/silicone, alginato, PHMB gel, membranas de biocelulose, NPWT
🔄
Fases da Cicatrização
Inflamação → Proliferação → Remodelagem. Critério de cronicidade: >3 semanas sem fechamento
🧹
Desbridamento
Cirúrgico, enzimático, autolítico, mecânico — contraindicações, membros isquêmicos
🥗
Nutrição Cicatricial
Albumina sérica, aminoácidos, vitaminas C/K/A/E, zinco por fase de cicatrização
👩‍⚕️
Atribuições da Equipe
Enfermeiro, técnico/auxiliar, médico — prescrição, execução, encaminhamentos, COFEN
📏
Morfologia e Mensuração
Comprimento × largura × profundidade, cavidades, tunelizações, técnica de SF 0,9%
📚 Fontes indexadas pelo sistema: Manual de Padronização de Curativos — SMS São Paulo (Jan/2021) · EPUAP/NPIAP/PPPIA 2019 · COFEN Res. 159 · protocolos locais da SMS Pelotas. O sistema só responde com base nesses documentos — nada é inventado.
03 — Por que o Sistema de Pelotas precisa de Embeddings?

Um profissional de saúde da rede municipal pode perguntar de muitas formas sobre o mesmo assunto. Os embeddings garantem que o sistema encontre a resposta certa — mesmo quando a pergunta usa linguagem popular, abreviações ou termos diferentes dos documentos indexados.

🔍
Busca por Significado
O sistema entende que "ferida preta e seca", "escara" e "necrose de coagulação" são o mesmo conceito — sem necessidade de correspondência exata de palavras.
📋
Protocolo de Pelotas Indexado
Os curativos padronizados da SMS-Pelotas, fluxos de solicitação de insumos, atribuições por categoria profissional e códigos SUPRI estão todos disponíveis ao sistema.
🎯
Resposta Precisa e Local
A IA recupera exatamente o produto e conduta certa para cada tipo de ferida conforme o que está disponível na rede pública de Pelotas — não protocolos genéricos da internet.
🛡️
Sem "Alucinação Clínica"
Sem embeddings, a IA poderia indicar curativos inexistentes na rede ou conduta incorreta. Com eles, ela só responde com base nos documentos reais do município.
04 — Como a Similaridade Funciona na Prática

Quando o profissional faz uma pergunta, ela também é transformada em embedding. O sistema Python compara matematicamente com os trechos indexados do protocolo e recupera os mais relevantes.

❓ Pergunta do profissional da UBS:

"Qual curativo usar em ferida com muito exsudato e cheiro ruim?"

→ Vira vetor: [0.2, 0.9, 0.8, 0.1, 0.7, −0.1, ...]

📄
Antimicrobianos (prata/PHMB) para feridas infectadas com exsudato intenso — Manual SMS-SP p. 49
93% similar ✓
📄
Diagnóstico de infecção: odor fétido, aumento de drenagem, tecido friável — protocolo Pelotas
81% similar ✓
📄
Escala Braden — avaliação de risco de lesão por pressão
12% similar ✗

✅ O sistema prioriza corretamente o conteúdo sobre infecção e curativos antimicrobianos e descarta o trecho sobre Escala Braden, irrelevante para essa pergunta.

05 — Pipeline Python Completo: RAG para o Sistema de Pelotas

RAG = Retrieval Augmented Generation — o sistema Python primeiro busca nos documentos do protocolo municipal indexados, depois gera a resposta com base no que encontrou. O resultado é uma orientação clínica fundamentada no protocolo real de Pelotas, não em dados genéricos da internet.

Fase 1 — Indexação (feita pela equipe de TI, uma única vez)

📁
Documentos
PDFs do protocolo municipal, Manual SMS-SP, tabelas SUPRI, fluxogramas
✂️
Chunking
Divide em trechos de ~900 tokens com sobreposição de 150 tokens
🧮
Embeddings
Modelo de embedding transforma cada trecho em vetor numérico de alta dimensão
🗄️
Vector DB
Vetores salvos no Chroma DB (servidor local da SMS Pelotas)

Fase 2 — Resposta em tempo real (cada consulta do profissional de saúde)

1
Profissional de saúde faz uma pergunta
"Quais curativos estão disponíveis na SMS Pelotas para úlcera venosa com exsudato moderado?"
💬 Input
2
Pergunta vira Embedding (Python)
O mesmo modelo de embedding usado na indexação transforma a pergunta em vetor compatível com o banco de dados
🧮 Embedding
3
Busca Semântica no Chroma DB local
O sistema encontra os 4–5 trechos mais similares do protocolo municipal, com score de relevância por similaridade de cosseno
🔍 Retrieval
4
LLM Gera Resposta Baseada no Protocolo
Recebe: pergunta + contexto clínico + trechos recuperados do protocolo de Pelotas → gera resposta citando o produto padronizado, código SUPRI quando aplicável, e orientações de uso
✨ Generation
5
Profissional recebe resposta com fontes e pode exportar
Resposta fundamentada no protocolo real, indicando o produto disponível na rede, frequência de troca, cuidados peri-lesionais e encaminhamentos quando necessário
📲 Output
06 — Com RAG vs. Sem RAG — Impacto Clínico Real

❌ Sem RAG (IA genérica)

"Para úlcera venosa, recomenda-se curativo de alginato de cálcio trocado a cada 2 dias..."
⚠ Pode indicar produto não disponível na rede de Pelotas, frequência errada ou conduta desatualizada

✅ Com RAG (protocolo de Pelotas)

"Conforme a padronização SMS-Pelotas, para ferida com exsudato moderado, indicar Biatan Alginato (cód. SUPRI XXXX). Trocar a cada 3 dias ou conforme saturação..."
✓ Produto real disponível na rede, com código de solicitação e frequência correta

✅ Como funciona na prática: Cada PDF e documento da base do sistema de Pelotas — Manual SMS-SP (2021), protocolos locais, tabelas SUPRI, fluxogramas de solicitação — é fragmentado em trechos, transformado em vetor pelo modelo de embedding e armazenado no Chroma DB (servidor local da SMS Pelotas). Ao receber uma pergunta, o sistema localiza os trechos mais relevantes por similaridade de cosseno e os entrega ao LLM para gerar uma resposta clínica precisa, fundamentada e rastreável.

🐍 Stack Python deste sistema: Embeddings → sentence-transformers ou OpenAI text-embedding-3-small  |  Geração → LLM local / API  |  Vector Store → chromadb  |  Orquestração → LangChain
